# مولّد امتحانات بأسلوب الدكتور — نظام عالي الدقة

## المعمارية الطبقية (v2 — الحالية)

النظام مُعاد هندسته حول مبدأ واحد: **الـ LLM مورد نادر — يُستدعى فقط لما لا
يمكن فعله بأدوات حتمية، وكل نداء يخدم دفعة كاملة لا سؤالاً واحداً.**
النتيجة المقاسة: امتحان 12 سؤالاً = **4 نداءات LLM** (كان ~30 في v1).

```
        ┌────────────────────── Orchestrator (حالة مشتركة + checkpointing) ─┐
        │                                                                    │
مدخلات ─► plan ─► generate ─► structure ─► dedup ─► cross_verify ─► repair ─► assemble
        │ حتمي    LLM دفعات     حتمي        حتمي       LLM دفعات      LLM      حتمي
        │                                                  ▲            │
        │                                                  └── ≤2 جولة ─┘
        └───── كل نداء LLM يمر عبر Gateway: تسلسل + تهدئة + ميزانية + سجل ──┘
```

- **plan** (حتمي): جدول مواصفات ببصمة الدكتور + تجميع الخانات دفعات
  متجانسة الموضوع.
- **generate** (LLM): نداء واحد يولّد 5-8 أسئلة بسياق مشترك (grounding +
  exemplars للدفعة كلها). حفظ الحالة بعد كل دفعة.
- **structure** (حتمي): فحص وتطبيع دفاعي (خيار صحيح واحد، لا تكرار خيارات...).
- **dedup** (حتمي): كشف تكرار داخل الامتحان + تسريب من بنك الدورات
  (تشابه نصّي/دلالي، عتبة 0.85).
- **cross_verify** (LLM): **التحقق المتقاطع بين نموذجين** — نموذج مختلف عن
  المولّد يحلّ كل الأسئلة دون رؤية المفاتيح (10 أسئلة/نداء). اتفاق ⇒
  verified؛ تعارض ⇒ يُسجَّل مع حجة المحقّق ويذهب للإصلاح.
- **repair** (LLM): كل الفاشلات بأسبابها في نداء واحد ⇒ إعادة التحقق
  الكاملة (بحد أقصى جولتين — لا حلقات لا نهائية).
- **assemble** (حتمي): الامتحان النهائي + تقرير نداءات شفاف لكل غرض.

**Checkpointing:** الحالة تُحفظ بعد كل عقدة وكل دفعة؛ انقطاع (429/ميزانية)
⇒ `--resume` يكمل من آخر نقطة دون إعادة الدفعات المنجزة (مُثبت باختبار).

```bash
python3 main.py generate --exams data/exams --lectures data/lectures \
    --style data/doctor_style_profile.json -n 30 --coverage \
    --gen-model mistral-large-latest --verify-model mistral-small-latest \
    --batch-size 6 --rate-delay 3 --llm-budget 20 -o exam.json
# انقطع؟ أعد نفس الأمر مع --resume
```

ملاحظة أكاديمية (اذكرها في التقرير): التحقق المتقاطع بين نموذجين يرفع
الثقة لكن لا يبرهنها — النماذج المختلفة قد تتشارك الخطأ نفسه لتشابه بيانات
تدريبها. التحقق بتنفيذ الكود (`code_verifier.py`) ما يزال متاحاً كطبقة
اختيارية، والمقارنة بين الاستراتيجيتين تجربة جاهزة لفصل النتائج.



نظام Python لتوليد أسئلة امتحانات (المنطق الصبابي والخوارزميات الجينية) تحاكي
أسلوب أستاذ محدّد، مع **تحقق حسابي مستقل** يضمن أن كل مفتاح إجابة صحيح فعلاً.

## لماذا هذا التصميم؟

المشكلة الأصعب ليست تقليد الأسلوب — بل **صحّة مفتاح الإجابة**. أي LLM يُنتج
سؤالاً بأسلوب رائع لكن بإجابة خاطئة حسابياً في نسبة معتبرة من الأسئلة الرياضية.
لذلك النظام مبني حول التأريض + التحقق، لا حول الإبداع الحر.

> برهان عملي: تدقيق بنك أسئلتك الحالي كشف أن مفتاح إجابة **Q1** خاطئ —
> `(A+A)(7)` بمبدأ التوسّع = **0.2** لا **0.6** كما هو مخزّن. شغّل `audit_bank.py`.

## المعمارية (7 طبقات)

1. **جدول المواصفات** (`blueprint.py`) — يوزّع الأسئلة على
   (topic × cognitive_level × difficulty) ليطابق البصمة الإحصائية للدكتور، قبل التوليد.
2. **استرجاع مؤرّض** (`retrieval.py`) — لكل خانة: grounding chunks من
   `normalized_knowledge` (المحتوى) + أمثلة أسئلة حقيقية مشابهة (الأسلوب).
3. **مولّد** (`prompts.py`) — يحاكي الأسلوب عبر أمثلة، ويُلزَم بإخراج
   `computation_spec` قابل للحساب الآلي.
4. **تحقق حسابي** (`fuzzy_math.py` + `verify.py`) — يعيد حساب الإجابة بكود فعلي
   ويرفض أي تعارض. **هذه أكبر رافعة دقة.**
5. **محقّق مغلق العينين** (`verify.py` L3) — LLM مستقل يحلّ السؤال دون رؤية المفتاح ويوافق.
6. **ضبط المشتّتات** — يتأكد أن كل مشتّت خاطئ فعلاً (لا إجابتان صحيحتان).
7. **كشف التكرار** — تشابه ضد البنك الأصلي لمنع إعادة إنتاج سؤال موجود.

القرار: `verified` (اجتاز الكل) / `needs_review` (مفاهيمي أو خلاف المحقّق) / `rejected` (خطأ حسابي).

## التحقق الهجين (compute_check + cross_verify)

بعد أول تشغيل حقيقي، كشف التدقيق اليدوي مفتاحين خاطئين مرّا عبر التحقق
المتقاطع وحده (خطأ مرتبط: النموذجان أخطآ نفس الحساب). الحل المطبَّق:

- **العقدة الحتمية compute_check** (بعد dedup، قبل cross_verify): تنفّذ
  `verification_code` المرفق بكل سؤال حسابي (ضمن نفس نداء الدفعة — صفر
  نداءات إضافية) وتقارن الناتج بالمفتاح. عند التعارض **تتبنى ناتج الكود**:
  1) الناتج يطابق خياراً آخر ⇒ قلب المفتاح إليه حتمياً فوراً.
  2) لا يطابق أي خيار ⇒ إرسال للإصلاح والناتج المحسوب مضمَّن في التعليمة
     (النموذج يصيغ فقط، لا يحسب).
  3) فشل تنفيذ الكود ⇒ يمر للتحقق المتقاطع وحده بلا امتياز.
- **الأسئلة المفاهيمية** (عرّف/قارن/نظري): `verification_code = null`
  بالتصميم — لا كود يُطلب ولا يُفتعل، ويحرسها كاشف "الكود الزائف" (كود
  يطبع ثابتاً بلا حساب يُتجاهل).
- سؤال حسابي verified الآن اجتاز **حَكَمين مختلفَي الطبيعة**: آلة حتمية
  نفّذت الحساب + نموذج ثانٍ حلّه مستقلاً.

**تدقيق الامتحانات الموجودة**: `python3 audit_generated_exam.py exam.json`
يفحص بنيوياً وحسابياً (إن وُجد كود) ويستخرج قائمة مراجعة يدوية للحسابي
القديم بلا كود.

## معالجة أسئلة 'غير ذلك' (NOTA) — إصلاح جذري لا منع

'غير ذلك'/'none of the above' كإجابة صحيحة **نمط مشروع قياسياً** عندما يكون
السؤال كامل المعطيات — المشكلة الحقيقية هي السؤال ناقص المعطيات الذي
"يرقّعه" النموذج بخيار جامع. المعالجة:

1. عند اكتشاف خيار جامع كإجابة صحيحة ⇒ يُرسل السؤال لعقدة الإصلاح بتعليمة
   صريحة: **أكمل المعطيات الناقصة** واجعل السؤال قابلاً للحل بإجابة محددة؛
   وإن كان كاملاً و'غير ذلك' مقصودة فعلاً، أبقِه مع حل كامل يثبت ذلك.
   (مرة واحدة لكل سؤال — لا حلقات مهدرة.)
2. إن بقيت 'غير ذلك' بعد الإصلاح **ووافق المحقّق المتقاطع** ⇒ تُقبل
   verified مع توسيم إعلامي فقط (`catch_all_correct: true`) ليقرر المدرّس.
   لا منع ولا هبوط تلقائي للحالة.

## استكشاف الأخطاء: JSONDecodeError عند نهاية توليد دفعة

عطل حقيقي واجهه أول تشغيل على بيانات فعلية: استجابة Mistral لدفعة (6
أسئلة) وصلت مقطوعة أو مشوّهة، فرمى `json.loads` خاماً وتحطّم التشغيل
بالكامل. السبب الجذري الأرجح: **`max_tokens` الافتراضي (2000) غير كافٍ
لدفعة كاملة**، خصوصاً بالعربية (كثافة توكن أعلى). تم إصلاح هذا على
مستويات متعددة:

1. **اكتشاف الانقطاع صراحة** (`finish_reason == "length"` من استجابة
   Mistral نفسها، لا تخميناً من خطأ JSON) + **تصعيد تلقائي**: إعادة
   المحاولة بـ max_tokens أكبر (×1.6، مرتين كحد أقصى) قبل الاستسلام.
2. **إصلاح JSON متعدد الطبقات** (`_extract_json`): فواصل زائدة، اقتباسات
   ذكية، ثم **إغلاق القطع الحقيقي** — يتتبّع مكدس الأقواس المفتوحة
   ويُغلق كل ما تبقى مفتوحاً يدوياً، فيستخرج أطول محتوى صالح بدل رمي كل
   الدفعة.
3. **رسالة تشخيصية واضحة** (`JSONExtractionError`) بدل Traceback خام —
   تقترح الحل مباشرة (`--max-tokens` أكبر أو `--batch-size` أصغر).
4. **تقسيم الدفعة الفاشلة بدل تحطيم التشغيل**: دفعة تفشل تُقسَّم نصفين
   وتُعاد المحاولة تلقائياً (حتى حجم 1)؛ خانة تفشل حتى منفردة تُسجَّل
   `rejected` بسبب واضح، ولا تُسقط بقية الامتحان. الأمر نفسه لعقدتي
   `cross_verify` و`repair`.

```bash
# إن تكرر الانقطاع رغم التصعيد التلقائي:
python3 main.py generate ... --max-tokens 6000 --batch-size 4
```

## استكشاف الأخطاء: KeyError عند نهاية توليد الخانات (options/text مفقود)

كانت مخرجات LLM (خصوصاً Mistral) أحياناً تُرجع خيارات بأسماء حقول مختلفة
(`option_text`, `content`, ...) أو بنية ناقصة، فيتحطم النظام بـ `KeyError`.
تم إصلاح هذا جذرياً:
- `verify.py`: طبقة تطبيع (`_normalize_options`) تتحمّل أسماء حقول بديلة
  وتسميات مفقودة، بالإضافة إلى شبكة أمان نهائية (`verify_question`) تحوّل
  أي خطأ غير متوقع إلى قرار "رفض" بدل تحطيم التوليد بالكامل.
- `pipeline.py` (`_to_schema`): دفاعي بالكامل، لا يفترض اكتمال حقول مخرجات LLM.
- `tests/test_robustness.py`: 9 اختبارات تغطي هذه الحالات (شغّلها بعد أي تعديل).

النتيجة العملية: سؤال واحد مشوّه الآن يُصنَّف `rejected` مع تفصيل واضح
لسبب الرفض، بدل إيقاف توليد بقية الامتحان بالكامل.

## استكشاف الأخطاء: 429 Rate limit exceeded / Connection reset by peer

شائع جداً مع مفاتيح الطبقة المجانية في Mistral Studio — لم تعد Mistral
تنشر أرقام الحد الأقصى للطبقة المجانية علناً؛ تحقّق من حدّك الفعلي عبر:
**https://admin.mistral.ai** → صفحة Limits.

`MistralClient` يتعامل مع هذا تلقائياً:
- يميّز 429/rate_limited عن انقطاع الاتصال العابر ويعطيه مهلة أطول (تبدأ
  من 5s وتتضاعف حتى `--max-retry-delay`).
- يحترم رأس `Retry-After` إن أرسله الخادم.
- القيم الافتراضية معدّلة للطبقة المجانية: `--rate-delay 3.0` (انتظار ثابت
  بين كل نداء) و`--retries 8`.

إن استمر 429 رغم ذلك:
```bash
# ابدأ بمهلة أكبر بين الطلبات (فعّالة جداً مع الطبقة المجانية):
python3 main.py generate ... --rate-delay 8.0

# ووسّع سقف التراجع الأُسّي لمحاولات 429:
python3 main.py generate ... --rate-delay 8.0 --retries 10 --max-retry-delay 90
```
لو استمر الخطأ حتى مع `--rate-delay 8.0`، فحدّك الحقيقي أقل من طلب كل
8 ثوانٍ — تحقّق من رقمه الدقيق في Admin Console، أو فكّر بالانتقال إلى
Scale plan (pay-as-you-go) الذي يرفع الحدود تلقائياً مع الاستخدام.

## التشغيل — واجهة سطر الأوامر (main.py)

```bash
STYLE=doctor_style_profile.json

# 1) توليد امتحان شامل من كامل المادة (مجلدات كاملة)
python3 main.py generate --exams data/exams --lectures data/lectures \
    --style $STYLE -n 40 --coverage -o exam.json
#   أضف --mock للتجربة دون شبكة | --no-blind لتسريع (أقل دقة)

# 2) تدقيق مفاتيح إجابة أي ملف أسئلة (يكشف المفاتيح الخاطئة)
python3 main.py audit --file exam.json

# 3) مطابقة الأسلوب كمياً (Jensen-Shannon divergence) — لفصل النتائج
python3 main.py evaluate --exams data/exams --lectures data/lectures \
    --style $STYLE --generated exam.json

# 4) دراسة الحذف: أثر طبقات التحقق على دقة المفاتيح — لفصل النتائج
python3 main.py ablation --exams data/exams --lectures data/lectures \
    --style $STYLE -n 15

# 5) مواد اختبار التمييز الأعمى (استمارة مشاركين + مفتاح سري)
python3 main.py blindtest --exams data/exams --lectures data/lectures \
    --style $STYLE --generated exam.json -o blind_test/
```

## وحدة التقييم الأكاديمي (exam_gen/evaluate.py)

- `style_fidelity(original, generated)` — يقارن توزيعات بلوم/الصعوبة/طول
  السؤال/عدد الخيارات بمقياس JSD (0 = تطابق تام).
- `build_blind_test(...)` + `analyze_blind_responses(...)` — تجربة التمييز
  الأعمى كاملة: دقة تمييز ≈ 50٪ (صدفة) تعني أن التقليد ناجح، مع z-test ضد الصدفة.
- `run_ablation(pipeline, n)` — الشرط A (توليد خام) مقابل الشرط B (كامل
  التحقق)؛ الفرق في دقة المفاتيح = الأثر المقاس لمساهمتك.
- `computational_key_accuracy(questions)` — دقة مفاتيح أي دفعة أسئلة آلياً.

### التشغيل الحقيقي مع LLM

في `run_demo.py` استبدل:

```python
from exam_gen import MistralClient
llm = MistralClient(model="mistral-large-latest")   # يتطلب: pip install "mistralai==1.5.1" + MISTRAL_API_KEY
```

الواجهة `LLMClient.complete_json(messages)` مجرّدة — يمكن ربط أي مزوّد (OpenAI، محلي، ...)

مرّر `embed_fn` إلى `ExamPipeline` لترقية الاسترجاع من كلمات مفتاحية إلى تشابه دلالي.

## توسيع المحرك الحسابي

كل عملية جديدة تُضاف في `fuzzy_math.evaluate_spec()` تصبح فوراً قابلة للتحقق.
العمليات المدعومة حالياً: membership, alpha_cut, scalar_cardinality, height,
support, core, union, intersection, complement (standard/sugeno/yager),
extension_binary (add/sub/mul), consistency, subsethood, defuzzify (centroid/mom)،
مع t-norms/t-conorms (min, product, lukasiewicz, drastic).

## المخرجات

بنفس بنية `all_exam_questions.json` (drop-in) مع حقول زيادة:
`computation_spec`, `provenance_chunk_ids`, `verification` (سجل الطبقات),
`worked_solution`. الأسئلة المفاهيمية غير القابلة للحساب تُوجَّه للمراجعة البشرية.
